## 📌 1. 기본 요구사항 (미니 레디스 구현)
- [x] **해시 테이블 활용 파이썬 키-값 저장소 직접 구현**
  - 👉 파이썬 내장 `dict` 자료구조를 활용해 메모리 직접 접근($\mathcal{O}(1)$) 인메모리 저장소를 구축했습니다.
  - 📍 **구코드 위치**: `server.py` **68번 줄** (`store: dict[str, Entry] = {}`)
- [x] **해당 저장소는 외부에서 사용할 수 있어야 함**
  - 👉 독립적인 TCP 서버 호스팅 소켓을 열어 외부 컴퓨터의 접속을 수용합니다.
  - 📍 **코드 위치**: `server.py` **739번 줄 부근** (`def run_server(host, port)`)
- [x] **핵심 로직 이해 및 동작방식 셜명 가능**
  - 👉 $\mathcal{O}(1)$ 해시간 접근, 락 원자성 등 주요 동작을 `README.md` 구조에 상세히 문서화 완료했습니다.

## 🔥 2. 중점 포인트 5가지 (가장 중요한 아키텍처 평가 기준)
- [x] **① 동시성 문제 방지 구조 고민**
  - 👉 수천 명이 동시에 한 좌석을 누르는 '이선좌' 문제를 방어하기 위해 단일 스레드 락(Lock)을 채택해 트랜잭션의 원자성을 보장했습니다.
  - 📍 **코드 위치**: `server.py` **69번 줄** (`store_lock = threading.RLock()`)
- [x] **② 보관 기간 만료(TTL) 데이터 처리 방안**
  - 👉 조회 시 지연 검사(Lazy Expiration)로 기한이 지난 데이터는 즉각 `None`을 뱉어내고, 백그라운드 스레드 가비지 컬렉터가 10초마다 순회하며 유저와 좌석을 쓸어담아 청소합니다.
  - 📍 **코드 위치**: `server.py` **120번 줄** (`get_visible_entry_locked()`), `server.py` **336번 줄** (`cleanup_ticket_event_locked()`)
- [x] **③ 외부 쉬운 사용을 위한 API 브릿지 설계**
  - 👉 복잡한 소켓 통신을 모르는 프론트엔드/클라이언트도 접근할 수 있도록 HTTP ↔ TCP (JSON ↔ RESP) 프로토콜 변환 브릿지 계층(API Gateway)을 자체 구축했습니다.
  - 📍 **코드 위치**: `client.py` **252번 줄 부근** (`class TicketingBridgeHandler`)
- [x] **④ 데이터 무효화(Deprecation/Invalidation) 방식**
  - 👉 실제 메모리에서 완전히 날려버리는 `Hard Delete` 방식과 더불어, 생명주기 객체에 취소선만 긋는 `임시 무효화(Invalidated_at)` 방식을 모두 구현했습니다.
  - 📍 **코드 위치**: `server.py` **38번 줄** (`Entry` 클래스의 `invalidated_at` 필드), **152번 줄** (`invalidate_key_locked()`)
- [x] **⑤ (Optional) 서버 다운 시 데이터 보존(Persistence)**
  - 👉 지정된 간격마다 `metadata.json` 파일에 메모리 해시맵 전체를 직렬화하여 덤프해두고, 무단 셧다운 재부팅 시 이를 복구(Load)하는 스냅샷 영속성(Persistence)을 구축했습니다.
  - 📍 **코드 위치**: `server.py` **96번 줄** (`save_snapshot_locked()`), **74번 줄** (`load_snapshot()`)
