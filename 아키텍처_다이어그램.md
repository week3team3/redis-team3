## 1. 🌐 전체 시스템 흐름도 (3-Tier Architecture)

사용자의 클릭이 어떻게 최종적으로 데이터를 변경하는지에 대한 **전체 데이터 여행 경로**입니다.

```mermaid
flowchart LR
    subgraph "Tier 1: 프론트엔드 (Presentation)"
        Browser["🖥️ 웹 브라우저\nindex.html + app.js"]
    end

    subgraph "Tier 2: 브릿지 서버 (Application)"
        Bridge["🌉 client.py\nTicketingBridgeHandler\nHTTP ↔ RESP 변환기"]
    end

    subgraph "Tier 3: 미니 레디스 코어 (Data)"
        Server["🗄️ server.py\nTCP 소켓 서버\nRLock 원자성 보장"]
        Store["📦 store: dict\n해시 테이블 저장소"]
        Sweeper["🧹 Sweeper 스레드\nTTL 만료 자동 청소"]
        Snapshot["💾 Snapshot\n디스크 영속성"]
    end

    Browser -- "HTTP JSON\n(POST /api/hold)" --> Bridge
    Bridge -- "TCP RESP\n(*3\\r\\n$12\\r\\nTICKET_HOLD...)" --> Server
    Server -- "RESP 응답" --> Bridge
    Bridge -- "JSON 응답" --> Browser

    Server --> Store
    Sweeper -- "주기적 순회" --> Store
    Server -- "주기적 덤프" --> Snapshot
    Snapshot -- "서버 시작 시 복구" --> Store
```

## 2. 🎫 좌석 예매 상태 전이도 (State Machine)

하나의 좌석이 거칠 수 있는 모든 상태 변화와 트리거(원인)를 보여줍니다.

```mermaid
stateDiagram-v2
    [*] --> AVAILABLE: 이벤트 초기화

    AVAILABLE --> HELD: TICKET_HOLD\n(사용자가 좌석 클릭)
    HELD --> CONFIRMED: TICKET_CONFIRM\n(결제 확정)
    HELD --> AVAILABLE: TICKET_CANCEL\n(사용자가 취소)
    HELD --> AVAILABLE: Hold TTL 만료\n(Sweeper가 자동 회수)

    CONFIRMED --> [*]: 예매 완료 (최종 상태)
```

## 3. 👤 사용자 입장/대기 상태 전이도

유저가 티켓팅 시스템에 진입할 때 거치는 상태 변화입니다.

```mermaid
stateDiagram-v2

    WAITING --> ADMITTED: 앞사람 이탈 시\npromote_from_queue

    ADMITTED --> HELD: 좌석 선점
    ADMITTED --> EXITED: TICKET_EXIT\n(자발적 이탈)
    ADMITTED --> EXITED: SESSION_IDLE_TTL 만료\n(10초 무반응 강제 퇴출)

    HELD --> CONFIRMED: 결제 확정
    HELD --> ADMITTED: 좌석 취소\n(입장 상태 유지)

    WAITING --> EXITED: SESSION_IDLE_TTL 만료

    CONFIRMED --> [*]: 예매 완료
    EXITED --> [*]: 시스템 이탈
```

## 5. 📡 요청 1건의 전체 여행 (Sequence Diagram)

사용자가 **좌석 Hold 버튼을 누르는 순간**부터 **화면에 결과가 표시**되기까지의 전체 여정입니다.

```mermaid
sequenceDiagram
    actor User as 👤 사용자
    participant UI as 🖥️ 웹 브라우저
    participant Bridge as 🌉 브릿지 서버
    participant Redis as 🗄️ 미니 레디스

    User->>UI: 좌석 A1 클릭
    UI->>Bridge: POST /api/hold (JSON)
    Bridge->>Bridge: JSON → RESP 직렬화
    Bridge->>Redis: TCP 전송 (RESP 바이너리)

    Redis->>Redis: store_lock 획득 (RLock)
    Redis->>Redis: 좌석 상태 확인 (AVAILABLE?)
    Redis->>Redis: seat:demo:A1 = HELD:user1 저장
    Redis->>Redis: TTL 설정 (Hold 만료 시간)
    Redis->>Redis: store_lock 해제

    Redis-->>Bridge: RESP 응답 (성공)
    Bridge->>Bridge: RESP → JSON 역직렬화
    Bridge-->>UI: JSON 응답
    UI->>UI: 좌석 색상 노란색으로 변경
    UI-->>User: 화면 업데이트 완료
```

## 6. ⏱️ TTL 만료 & Sweeper 동작 흐름

백그라운드에서 조용히 돌아가며 만료된 데이터를 청소하는 **Sweeper 스레드**의 동작입니다.

```mermaid
sequenceDiagram
    participant Sweeper as 🧹 Sweeper 스레드
    participant Store as 📦 Hash Table
    participant Queue as 📋 대기열 ZSet

    loop 매 1초마다
        Sweeper->>Store: store_lock 획득
        Sweeper->>Store: 모든 세션 키 순회
        alt 세션 TTL 만료 (10초 무반응)
            Sweeper->>Store: 유저 세션 키 삭제
            Sweeper->>Store: 입장자 목록에서 제거   
            Sweeper->>Store: Hold 중인 좌석 반납 (AVAILABLE)
            Sweeper->>Queue: 대기열 1순위 유저 승격 (ADMITTED)
        end
        alt Hold TTL 만료 (결제 안 함)
            Sweeper->>Store: 좌석 상태 AVAILABLE로 복원
            Note over Sweeper,Store: 유저의 입장(ADMITTED) 상태는 유지
        end
        Sweeper->>Store: store_lock 해제
    end
```
