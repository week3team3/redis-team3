# Mini Redis Ticketing: 웹 UI 시연 시나리오

이 문서는 새롭게 개편된 '티켓링크' 스타일의 웹 UI를 활용하여, 프로젝트의 핵심 기능(상태 관리, 대기열, TTL, 동시성)을 단계별로 보여주기 위한 시연 대본(가이드)입니다. 

---

## 🚀 시연 준비
1. 터미널 창을 두 개 열어 백엔드와 프론트엔드 브릿지 서버를 각각 실행합니다.
   - `python3 server.py` (Redis 서버 구동)
   - `python3 client.py` (웹 브릿지 서버 구동)
2. 브라우저에서 `http://127.0.0.1:8000`에 접속합니다.

---

## 🎬 시나리오 1: 기본 예매 및 결제 흐름 (Happy Path)
**목표**: 좌석을 선택하고 확보(Hold)한 뒤 최종 결제(Confirm)하는 정상적인 과정을 보여줍니다.

1. **초기화**: 우측 '이벤트 설정' 패널에서 **[이벤트 초기화]** 버튼을 클릭합니다. (기본 설정 적용)
2. **입장**: '수동 사용자 조작' 패널에서 User ID를 `user-a`로 입력하고 **[예매 입장]**을 클릭합니다.
   - 👉 우측 하단 **[입장자]** 리스트에 `user-a`가 표시되는지 확인합니다.
3. **좌석 Hold**: 중앙의 좌석 맵(Seat Map)에서 빈 좌석(예: FLOOR 구역의 첫 번째 좌석)을 클릭합니다.
   - 클릭 후 바로 **[선택 좌석 Hold]** 버튼을 누릅니다.
   - 👉 좌석 색상이 보라색(Available)에서 노란색(Held)으로 즉시 변경됨을 확인합니다.
4. **결제 확정**: 이어서 **[Confirm]** 버튼을 누릅니다.
   - 👉 좌석 색상이 회색과 X표시(Sold)로 변경되며 예약이 확정되는 것을 보여줍니다.
5. hold 다시 캔슬하는거

---

## 🎬 시나리오 2: 대기열(Queue) 시스템 동작
**목표**: 동시 접속자가 정해진 인원(Active Limit)을 초과할 때, 대기열 큐로 넘어가고 기존 입장자가 빠지면 자동 승격되는 과정을 보여줍니다.

1. **초기화 (설정 변경)**:
   - '이벤트 설정' 패널에서 **Active Limit**을 `2`로 설정하고 **[이벤트 초기화]** 버튼을 누릅니다.
2. **접속자 3명 입장**:
   - User ID에 `user-1` 입력 -> **[예매 입장]**
   - User ID에 `user-2` 입력 -> **[예매 입장]**
   - User ID에 `user-3` 입력 -> **[예매 입장]**
3. **대기열 확인**:
   - 👉 우측 하단 패널을 보면 `user-1`, `user-2`는 **[입장자]**로, `user-3`은 **[대기열]** 리스트에 들어간 것을 강조합니다.
4. **기존 입장자 이탈 및 자동 승격**:
   - User ID를 다시 `user-1`로 바꾸고 **[입장 이탈]** (또는 예매 취소) 버튼을 클릭합니다.
   - 👉 1초 뒤 서버 Refresh 시, **대기열에 있던 `user-3`이 자동으로 '입장자' 목록으로 승격(Promoted)** 되는 마법을 보여줍니다.

---

## 🎬 시나리오 3: 타임아웃(TTL) 기반 좌석 회수 무효화
**목표**: 선택한 좌석을 제시간에 결제하지 않으면, Redis TTL 기능에 의해 좌석이 자동으로 풀리는 과정을 보여줍니다.

1. **초기화 (TTL 변경)**:
   - '이벤트 설정' 패널에서 **Hold TTL**을 짧게 `8`초로 설정하고 **[이벤트 초기화]**를 누릅니다.
2. **좌석 Hold 이후 방치**:
   - User ID `user-a` 상태로 **[예매 입장]**을 누릅니다.
   - 좌석 하나를 클릭하고 **[선택 좌석 Hold]**를 누릅니다.
3. **타임아웃 관찰**:
   - 좌석이 노란색(Held)으로 변한 것을 확인한 뒤 **Confirm을 누르지 않고 기다립니다!**
   - 👉 약 8초 후, 서버의 백그라운드 스윕(Sweep)에 의해 TTL이 만료되면 **좌석이 다시 보라색(Available) 공석으로 짠 하고 돌아오는 처리**를 시각적으로 보여주며 설명합니다.

---

## 🎬 시나리오 4: 스레드 동시성(Safe Concurrency) 검증 시뮬레이션 (랜덤좌석 시뮬레이션)
**목표**: 백화점 오픈런이나 인기 콘서트 티켓팅에서 발생하는 '이선좌(이미 선택된 좌석입니다)' 동시 접근 문제를 단일 서버에서 어떻게 안전하게(Safe) 처리하는지 보여줍니다.

1. **초기화**: 우선 쾌적한 맵을 위해 **[이벤트 초기화]**를 한 번 눌러줍니다.
2. **시뮬레이션 설정**: 
   - 우측 패널에서 시뮬레이션 **Users**를 `40`명으로 설정합니다.
   - **Scenario**를 `같은 좌석 경쟁(same-seat)`으로 맞춥니다.
   - 맵에서 목표가 될 타겟 좌석을 하나 일부러 클릭하여 선택해둡니다.
3. **부하 테스트 실행**:
   - **[시뮬레이션 실행]** 버튼을 클릭합니다.
4. **결과 확인 (중요)**:
   - "40개의 쓰레드가 동시에 하나의 좌석을 요청했습니다" 라고 설명합니다.
   - 👉 우측 로그 스트림과 시뮬레이션 결과에 `users=40, confirmed=1, hold_fail=39` 가 찍힌 것을 강조합니다.
   - 오직 1명만 결제에 성공하고 39명은 오버셀링 없이 정상적으로 튕겨져 나갔음(안전성 입증)을 핵심 셀링 포인트로 설명합니다.
